### 參與者說明

1. 社團幹部是本次最主要的參與者，會直接操作申請表單、送出申請及查看狀態。
2. 課外活動組老師會使用系統查看待審核資料，但完整核准、退回與補件規則尚未確認。
3. 場地管理員需要場地使用資訊，但通知時機與方式尚未確認，因此暫時只列入分析，不列入本次程式實作。
4. 一般學生雖可能查看活動資訊，但直接訪談來源不足，暫時列為低優先、待確認參與者。
5. 資料庫、申請表單、活動列表及衝突檢查模組均屬於系統內部，不是參與者。

### F. 使用案例描述

使用案例編號：UC-01

使用案例名稱：建立活動申請

目標：
社團幹部建立活動申請，並送出完整的活動資料。

主要參與者：
社團幹部

支援參與者：
系統

觸發事件：
社團需要申請舉辦活動。

前置條件：

- 使用者已登入系統。
- 已具有活動申請權限。

成功後置條件：

- 系統建立新的活動申請。
- 活動狀態為「待審核」。

最低保證：
若建立失敗，不會新增任何活動資料。

主要流程：

1. 社團幹部開啟活動申請頁面。
2. 輸入活動名稱、日期、時間及場地。
3. 系統檢查必填欄位。
4. 系統檢查場地是否衝突。
5. 系統建立活動申請。
6. 系統更新狀態為待審核。
7. 顯示建立成功訊息。

替代流程：

- 編號：A-01
- 分支點：步驟4
- 條件：不同場地或不同時段。
- 步驟：系統允許建立活動申請。
- 結束或返回位置：返回步驟5。

例外流程：

- 編號：E-01
- 分支點：步驟4
- 條件：場地衝突。
- 系統處理：顯示場地已被借用，禁止送出。
- 結束狀態：返回活動申請畫面。

業務規則：
同一場地於相同時間不可重複借用。

輸入資料：
活動名稱、日期、時間、場地、活動說明。

輸出資料：
活動申請紀錄。

狀態變化：
建立中 → 待審核。

對應需求：
FR-01、FR-02、FR-03、FR-05、FR-06。

對應驗收條件：
AC-01-01、AC-02-01、AC-03-01。

待確認問題：
是否需要加入指導老師簽核流程。
